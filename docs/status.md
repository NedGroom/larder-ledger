Supporting infra
- API Gateway / ALB for routing; ALB for HTTP(s) and WebSocket if you keep persistent backend. Alternatively, API Gateway WebSocket + Lambdas (serverless) if you want fully serverless sockets (more complex routing).
- Cognito (or Auth0) for user auth; or JWT signed keys rotated securely with KMS/SecretsManager.
- CDK manages stacks: VPC, ECS cluster, RDS, S3, Redis, IAM roles, ALB, Route53/ACM, and optional MSK cluster.
- GitHub Actions build -> push image to ECR -> CDK deploy pipeline.

Why this is recommended
- ECS Fargate + ALB gives a straightforward way to run FastAPI + WebSocket with single persistent tasks and autoscale. Use Redis for cross‑instance broadcast so WebSocket connections on different tasks get events delivered.
- RDS provides durable relational storage and is compatible with Alembic migrations and your SQLAlchemy models.
- S3 + Textract (or Lambda workers) handles receipts at scale; local Tesseract remains as dev fallback.
- Kafka (MSK) is optional for long-term analytics/event sourcing; it is not required for MVP.

Gaps and risks
- Current in‑process WebSocket manager will not scale to multiple backend tasks. Add Redis pub/sub or a central message broker before multi‑AZ deployment.
- No authentication implemented yet — public endpoints exist until auth is added. Plan to add JWT/Cognito immediately before public deployment.
- Migration state: some alembic migrations assume tables exist (because create_all was used during development). For production, rebase and create clean baseline migrations and do a verified upgrade path.

LarderLedger — Short Introduction & Outline

Project summary
- Working name: LarderLedger
- Purpose: Mobile‑first PWA to help a small household (3 people) track pantry inventory, register meals, and generate shopping lists with per‑store price comparisons and outlier highlighting.
- Principles: Keep It Simple, Ship Fast. Start with the minimum infrastructure needed and extend only when required.

Target audience
- Housemates who cook and share groceries.
- Engineers/stakeholders evaluating feasibility and initial implementation.

MVP scope (short)
- Inventory: add ingredients, support multiple units, track optional quantities, and a boolean `has_any` flag to indicate whether the house has any of that ingredient.
- Meals: register meals with name, dish type (salad, veg side, protein+veg, full meal, dessert), prep time, servings, tags, chef tag, and price‑per‑portion.
- Shopping list: autogenerated missing ingredients (toggle to blend shopping list items into the full ingredient list); allow manual add/remove; simple checklist UI.
- Prices: per‑ingredient per‑store price fields (stores are user‑defined). Manual entry is primary; batch entry via receipt/photo upload is supported as a reviewable candidate list.
- Price analysis: for a selected view (cupboard | missing | shopping list) and a selected store, show per‑item prices and highlight price outliers versus median; optional on‑demand store‑level aggregate score.
- Collaboration: multi‑user with realtime updates for inventory and shopping lists; users can tag chef and filter meals.

Non‑goals for MVP
- No structured substitution engine (deferred).
- No automated scraping of store websites (deferred).
- No heavy streaming infra (no Kafka in MVP).
- Offline‑first is future work; read‑only cache is acceptable initially.

High‑level architecture (MVP)
- Frontend: Mobile‑first web app / PWA (React + TypeScript suggested) for fastest multi‑device deployment.
- Backend: Python FastAPI served by uvicorn. REST APIs + WebSocket for realtime.
- DB: PostgreSQL for relational data; S3 for receipt images. Use AWS CDK for infra provisioning in production; local dev via docker‑compose.
- OCR (optional): AWS Textract for production; Tesseract local fallback for development.

Success criteria (MVP)
- New users can register a house, create ingredients and meals, and generate a shopping list within 10 minutes of starting.
- Realtime updates are visible to other house members within a few seconds.
- Price outliers are clearly highlighted per‑store on demand.

Next steps
